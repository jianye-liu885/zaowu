# 笔试说明文档 · 造物 Zaowu

- **在线体验**：https://zaowu.wenloom.com （注册任意账号即可，已配好模型额度）
- **源代码**：本仓库
- **一句话**：一个 Atoms 形态的 Demo——对话驱动 Agent 生成完整 Web 应用，沙箱实时预览，可迭代、可回滚、可发布分享。

## 一、实现思路与关键取舍

**先把题目拆成一条不可妥协的主线**：`自然语言 → Agent 生成应用代码 → 生成的应用以网页形式真实运行`。所有取舍都围绕"这条主线要稳、要快、要可验证"。

| 决策点 | 选择 | 取舍理由 |
|---|---|---|
| 生成产物形态 | 单文件 HTML（CSS/JS 内联、零外部依赖） | 单文件 = 可靠渲染 + 可靠存储 + 可靠分享。多文件工程化产物（如生成 React 项目再构建）演示效果更炫，但引入构建链路和失败面，6-8 小时内稳定性优先 |
| 预览方案 | sandboxed iframe `srcdoc` | 零延迟、零服务端开销；`sandbox` 属性隔离生成代码，防 XSS 到宿主 |
| 迭代修改 | 全量重生成（当前代码+指令→新完整文件） | 备选是 diff/patch 式修改，token 更省但补丁错位风险高。全量重生成配合版本留存，错了就回滚，简单可靠 |
| 上下文策略 | 只带最新版本代码 + 近 8 条说明性对话（不带历史代码） | 控制上下文成本，同时保证"在现有基础上改"的正确性 |
| 持久化 | SQLite | 单机 Demo 规模下最合理选择，四张表（用户/项目/版本/消息）覆盖全部需求；换 MySQL 只需改连接层 |
| 后端形态 | FastAPI 单文件（~370 行） | Demo 的复杂度控制：一个文件讲清全部逻辑，评审阅读成本最低 |
| 流式体验 | SSE 逐字推送，代码围栏前的说明实时上屏，代码部分只显示计数 | 用户要的是"感觉到它在造"，不是看 3 万字代码刷屏 |

## 二、当前完成程度

**已完成（全部真实可用，非 PoC）**：
- ✅ 注册 / 登录（PBKDF2 加盐；token 鉴权）
- ✅ 核心主流程：新建项目 → 对话生成应用 → 沙箱实时预览 → 继续对话迭代修改
- ✅ SSE 流式直播（生成说明实时显示 + 字数进度）
- ✅ 版本管理：每次生成自动存版本，任意版本查看、一键回滚（回滚生成新版本，不破坏历史）
- ✅ 延展能力①——**发布分享**：一键生成公开短链，任何人无需登录直接使用该应用，可取消发布
- ✅ 延展能力②——**代码视图**：随时查看生成的完整源码
- ✅ 数据持久化（用户/项目/版本/对话全量落库）
- ✅ 生产部署：https + systemd 守护 + nginx（SSE 关缓冲）

**没做（识别到但主动放掉）**：
- ❌ 多文件/工程化项目生成（见取舍表第一条）
- ❌ 生成失败自动重试链（当前解析失败会给出明确错误提示，用户重发即可）
- ❌ 模板库 / 示例画廊（首页有 5 个一键起步的想法胶囊，算轻量替代）
- ❌ 协作/多人（超出 Demo 范围）

## 三、如果继续投入，扩展方向与优先级

1. **P0 · 局部修改（diff 补丁模式）**：小改动（换个颜色）不应重生成 3 万字。让模型输出结构化补丁，失败自动降级为全量重生成——成本降一个数量级，速度显著提升。
2. **P0 · 生成质检闭环**：生成后自动在无头浏览器里跑一遍（能否渲染、有无 JS 报错），不过关自动带错误信息回炉重写。这是从"能生成"到"生成得可靠"的关键一步。
3. **P1 · 多文件项目 + 依赖**：产物升级为 React/Vue 工程，服务端沙箱构建，覆盖更复杂应用。
4. **P1 · 数据后端托管**：给生成的应用提供 KV 存储 API（目前用 localStorage），分享出去的应用就有了真正的多用户数据。
5. **P2 · 应用广场**：已发布应用的公开画廊 + 一键 fork 到自己账号再改——形成"生成→分享→再创作"循环。
